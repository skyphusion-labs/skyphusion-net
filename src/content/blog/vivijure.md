A while back I wrote up [skyphusion-llm-public](/blog/llm/), a multimodal AI playground that lives entirely in one Cloudflare Worker. This post is about its other half: **Vivijure**, an AI film/video pipeline, and specifically `vivijure-serverless`, the GPU backend that does the actual rendering. Source is MIT at [github.com/SkyPhusion/vivijure-serverless](https://github.com/SkyPhusion/vivijure-serverless).

The pitch: you plan a film in a chat UI, the system trains a LoRA for each character so they stay consistent across shots, renders SDXL keyframes and turns them into motion with image-to-video, and hands you back a silent MP4. The render runs on a GPU that **scales to zero**; you pay only while a job is actually running, and $0 the rest of the time.

## The two-part system

Vivijure is two programs that meet at a contract.

**skyphusion** ([`skyphusion-llm-public`](https://github.com/SkyPhusion/skyphusion-llm-public), the subject of the [last post](/blog/llm/)) is the always-on control plane. It's a Cloudflare Worker, so it's always reachable and costs nothing at idle. It does the cheap, interactive, CPU-bound work: storyboard planning (chat your way to a shot list), cast prep (generate character reference images), and the light pre-processing (background removal, audio beat detection) that runs on Cloudflare Containers. When the plan is final, it assembles a **project bundle** (a `tar.gz` of the storyboard, the character training images, the reference frames) and writes it to Cloudflare R2.

**vivijure-serverless** (this repo) is the GPU backend. It's a [RunPod](https://runpod.io) serverless endpoint. A job arrives carrying an R2 key; the worker pulls the bundle, trains the character LoRAs, renders, exports the MP4, pushes the result back to R2, and returns the keys. Then the worker spins down to zero.

The split is the whole point. The two halves have completely different cost shapes. Planning is bursty, interactive, and nearly free to run; it wants to be always-on. Rendering is GPU-bound, runs for tens of minutes, and is expensive per-second; it wants to *not exist* when nobody's rendering. Putting them in one box means either paying for an idle GPU 24/7 or making your planning UI cold-start a GPU. Splitting them lets each half be cheap in its own way: the Worker is always up at $0 idle, and the GPU bills only for the minutes a render is in flight.

```
bundle (R2)  →  train LoRAs  →  SDXL keyframes + InstantID  →  image-to-video  →  silent MP4  →  R2
```

The trust boundary falls out of the split cleanly. The GPU backend only ever talks to **R2** (pull the bundle, push the results) and answers the RunPod endpoint with R2 keys. It never calls the control plane, never sees Cloudflare Access, never holds a skyphusion secret. The only credential on the GPU worker is one R2 API token scoped to one bucket. Everything the render needs is in the bundle; everything it produces goes back to R2. That's the entire conversation.

## The serverless layer is additive

Vivijure started life as a persistent GPU pod: FastAPI, tmux, a Cloudflare tunnel, a render loop driven by an HTTP UI. The serverless version doesn't reimplement any of that. It *wraps* it.

This repo owns exactly four files of harness:

- `rp_handler.py`: the RunPod entrypoint. Per job: prepare the volume, pull the bundle, run the orchestrator (forwarding progress back to the job), push the MP4 and updated state, return the keys.
- `orchestrator.py`, the `run_project()` function: train the ready character slots, render, export. That's the entire pipeline in one function.
- `r2_io.py`: S3-compatible get/put against R2 via boto3, with a path-traversal guard.
- `prime.sh`: a one-shot script to seed a fresh volume with model weights.

Everything else is the existing Vivijure render pipeline, untouched in spirit. The orchestrator calls the same blocking functions the old FastAPI UI called (`studio_service._render_worker_body(payload)`, `studio_service.export_film(name)`, `lora_train.train_character_lora(...)`); it just calls them *directly*, skipping the threading, the render lock, and the `render_status.json` polling that only existed to keep an HTTP request alive while a render ran in a background thread. Serverless doesn't need any of that: RunPod's queue-based endpoint *is* the "submit and poll" machinery, so the wrappers that reimplemented it in-process are dead weight.

The discipline this buys: pipeline behavior (model wiring, render logic, LoRA training) has exactly one home, the pipeline source. The harness never grows its own copy of render logic that drifts from the real thing. The closest I have to a regression test leans on this directly: a CPU-only `import orchestrator, r2_io` that pulls the entire transitive chain (`orchestrator` → `studio_service` → `core` → the video backends, `diffusers`, and so on) without launching the worker. A clean `imports OK` means the harness is wired up. No GPU required to know the build is sound.

## State round-trips through R2, not the volume

The non-obvious architectural call: the GPU's network volume holds **only immutable model weights**. Nothing the render produces lives there.

The naive design attaches a persistent volume and keeps everything on it: model weights, project trees, trained LoRAs, rendered frames. It's the obvious move and it's wrong for scale-to-zero. A serverless endpoint can run your job on a *different physical worker* every time. Anything you wrote to "the volume" last job might be on a machine you'll never be scheduled on again, and concurrent jobs on a shared volume race each other. So the volume is the one thing that's genuinely shared and read-only: the SDXL and Wan weights, the InstantID models. Populate it once, never write to it at run time.

Everything mutable round-trips through R2 instead. Each job:

1. Restores `projects/<name>/state.tar.gz` from R2 (the project tree plus any LoRAs trained on a prior render of this project),
2. Applies the input bundle on top,
3. Runs the render,
4. Uploads the updated project tree back as the next `state.tar.gz`.

R2 is the source of truth across jobs; the worker's local disk is scratch. This is the same split the LLM playground uses (structured pointers in one durable store, bytes in object storage), just pushed down to the render layer. And it's what makes the render *resume-aware* across the scale-to-zero gap: re-submit a tweaked project and it retrains no LoRAs and regenerates only the missing keyframes or clips, even though the worker that did the first render is long gone.

One small tar gotcha worth the warning, because I got it wrong once: the state tarball is packed contents-at-root (`arcname="."`), not `<name>/`-rooted, because the extractor already lands files inside the project directory. A name-rooted tar double-nests into `projects/foo/foo/...` and the next render quietly can't find anything.

## LoRA reuse: do the slow thing once

Training a character LoRA is the slowest part of a cold render (minutes of GPU time per character). On a per-second GPU bill, that's the line item worth attacking, so the worker skips training whenever a usable LoRA already exists. Two paths:

**Per-project state.** The `state.tar.gz` restored at the top of each job carries the trained `.safetensors` plus the `lora_train_<SLOT>.json` record the train-skip gate checks. So the second and later renders of a project see each character as already-trained and short-circuit straight to rendering. The first render of a given project always pays the training cost; every render after it doesn't.

**Pretrained passthrough (cross-project).** The job payload can map a character slot to an R2 key of an already-trained LoRA, say one trained standalone in skyphusion's cast manager. The worker stages it into the project and synthesizes the same job-json the skip gate looks for, so even a brand-new project reuses an existing character without retraining. The control plane resolves these keys from its cast database before it ever submits the job.

The effect is that a character you've trained once follows you across films for free. Cold renders are minutes-of-training expensive; warm renders skip straight to the GPU work you actually wanted.

## Two of everything, on purpose

Two things in this repo look like duplication a tidy-minded person would want to collapse. Both are load-bearing.

**Two mount points for one volume.** The network volume mounts at `/runpod-volume` on a serverless endpoint and at `/workspace` on a regular pod. That's a RunPod platform fact, not a choice. The priming script runs on a throwaway *pod* and writes weights to `/workspace/...`; the serverless *worker* reads the identical layout under `/runpod-volume`. Every path that joins a volume-relative location goes through a `WV_VOLUME` base so the same code works from both mounts. Hardcode either path and priming silently writes somewhere the worker never reads.

**Two CUDA targets.** The worker image is `nvidia/cuda:12.8.1` with `torch 2.7.* cu128`; the prime image is `nvidia/cuda:12.4.1`. The newer stack is non-negotiable for the worker because the GPUs are Blackwell (RTX PRO 6000, `sm_120`), and older cu124 builds raise `no kernel image is available for execution on the device` on those cards: a runtime failure, deep into a job, that looks like a model bug until you recognize it. The prime image can stay on the older CUDA because all it does is download and validate weights; it never renders. Trained `.safetensors` are portable across the gap, so the mismatch is safe. The rule I left in the docs: never *downgrade* the worker to match the prime image.

The meta-lesson, which the LLM post also kept running into: "this looks redundant, let me unify it" is a great instinct that's wrong roughly as often as it's right. When two near-identical things exist, the question isn't "can I merge them" but "what forced them apart in the first place." Here the answers are platform mount conventions and GPU architecture, neither of which a refactor can argue with.

## The silent-failure lessons

The [LLM post](/blog/llm/) had a RAG bug where try/catches swallowed the real error and every query returned an empty array, so a hard failure masqueraded as a relevance problem for weeks. The render pipeline taught me the same lesson from two more angles.

**Garbage-in from an over-eager LLM.** The render historically re-planned each scene from raw shot-list text using an LLM, right before drawing it. That re-planning step was the single biggest source of garbage in early runs: the control plane had already produced a careful storyboard, and then the render would creatively reinterpret it. The fix is a one-line seam: the orchestrator sets `prefinalized_scenes = True` so the render consumes skyphusion's finalized `storyboard.yaml` directly and skips re-planning entirely. The planning happens once, in the half of the system built to plan, and the render just executes. Every LLM call you *remove* from the hot path is a class of nondeterministic failure you delete with it.

**Failures that erase their own evidence.** When the render produced zero clips, the terminal RunPod payload would surface a bare `Missing clip`, and drop the entire progress log that explained *why*. The actual cause (a model gate tripped, a backend unavailable, weights missing from the volume) was gone by the time the job reported back, which meant every failure became a rebuild-and-redeploy guessing game. So there's now a diagnostics path that fires specifically when zero clips come out: it dumps the render-log tail, probes the volume and HF cache, checks backend availability, and validates the weight repos, all into the job output, before the worker dies. It exists purely so a failure explains itself on the first try instead of the third.

Both are the same principle the RAG bug hammered home: the expensive failures aren't the loud ones, they're the ones that look like success or look like a different problem. On a scale-to-zero GPU the feedback loop is brutally slow (spin up, run for minutes, fail, spin down), so paying up front to make failures legible is the highest-leverage code in the repo.

## A few smaller things worth knowing

**Output is silent by default.** Vivijure exports a silent MP4 and leaves audio to a downstream `ffmpeg` step. That's deliberate: it keeps the GPU pipeline focused on picture, and it sidesteps the audio-baking quirks some image-to-video backends have. Real shots route through the render pipeline; music and dialogue get muxed on afterward where they're cheap to redo.

**Multi-character shots.** When more than one character appears in a shot, a `regional` engine drives each one with its own LoRA and IP-Adapter inside a masked region of the frame. That masking keeps the figures spatially distinct, and each character's identity holds once the per-slot LoRA and IP-Adapter scales are tuned for the shot (the Aria and Martha render above is what under-tuned scales look like: the second character's identity drifted and swapped gender, a funny result I left in). Those scales are tunable per job, as is nearly everything else: the render takes typed `render_overrides` in the job payload, so the control plane can retune a render (art style, guidance, character placement) without rebuilding the image.

**Weights aren't in the image.** The Docker image is lean because the multi-gigabyte model weights live on the volume, primed once, not baked into a layer. The image holds no secrets either (R2 credentials arrive as endpoint environment variables at run time), so it's safe to host public on GHCR, and cold starts stay fast.

**Losing the network volume (where this is headed).** The network volume is the one always-on piece of an otherwise scale-to-zero design, and the priming step that seeds it is the price of keeping it. It also turned out to carry a cost I hadn't priced in: a network volume pins the endpoint to the specific datacenters that hold it, and those datacenters don't always have a free GPU when you want one. So the plan is to drop the persistent volume and let the pod's disk be ephemeral. Given a Hugging Face token, a cold worker pulls the model weights straight from the Hub over RunPod's fast links to their high-end GPU pods, which turns a cold prime into roughly five minutes of download instead of a standing storage bill. The trade is paying that download once per cold start, in exchange for lower storage cost and the freedom to land on whatever datacenter actually has capacity that day.

## What this is and isn't

This is the GPU half of a personal project I use to make AI films and videos, made public so people can fork it. It's not a SaaS and not a product. What it *is*, if you squint, is a fairly clean template for a particular shape of problem: a cheap always-on control plane (Cloudflare Workers) driving an expensive scale-to-zero compute backend (RunPod GPU), with object storage (R2) as the only thing they share and the only trust boundary between them. That pattern generalizes well past video rendering: any time you have bursty, expensive, stateless compute hanging off an interactive front end, the split and the R2-as-contract idea carry over.

The most useful single takeaway is the same one from building the LLM playground: let each tier be good at its own cost shape, and make the boundary between them boring. The Worker is always up and free at idle. The GPU is absent until summoned and bills by the minute. R2 is the dumb, durable thing in the middle that neither one has to trust the other to use. None of those pieces had to drop a layer to cooperate, which is exactly the kind of integration that's easy to undervalue until you've tried to build it on a stack where every piece comes from a different vendor.
